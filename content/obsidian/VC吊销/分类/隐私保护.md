
## 一、六个隐私风险

### 风险1：持有者与验证方关联泄露（Phone Home问题）

**注册表如何存储信息：**

在传统方案（如OCSP）中，发行方维护一个在线状态服务器。当验证方收到持有者的VC后，需要向发行方的服务器发送一个查询请求，请求内容包含该VC的唯一标识符（credential ID）。这个请求实际上告诉发行方："有人正在使用这张凭证"。

**第三方如何观察：**

发行方的服务器日志会记录：

- 查询时间戳
- 请求来源的IP地址（即验证方的地址）
- 被查询的凭证ID（对应特定持有者）

**如何推断关联关系：**

发行方将凭证ID与持有者身份对应（因为是自己发出去的），又知道是哪个验证方在查询，于是可以得出结论："持有者X在时间T向验证方Y出示了凭证"。在Prevoke论文的例子中，雇主1（发行方）得知雇主2（验证方）正在查询Bob的就业证明，从而推断Bob在秘密找工作。

---

### 风险2：持有者身份泄露（你重点问的部分）

**注册表如何存储这些信息：**

以IOTA Identity（位图方案）为例，其工作方式如下。在发行阶段，发行方给每张VC分配一个位图中的固定索引位置，例如Bob的VC对应第1000位，Alice的VC对应第1001位。这个索引号被写入VC本身，持有者知道自己的索引是多少。注册表（公开账本）上存储的是一个完整的位串，例如：

```
位置: 0    1    ...  1000  1001  ...
值:   0    0    ...  1     0     ...
```

当Bob的VC被吊销时，发行方将第1000位从0翻转为1，并将更新后的位串发布到公开账本。

**第三方如何观察到这些数据：**

由于账本是公开的，任何人都可以下载完整的位串，并且可以对比前后两个版本。观察者下载到两个时间点的位串后，做一个简单的差值运算：

```
旧版本: ...0 0 0 0 0...（第1000位为0）
新版本: ...0 0 1 0 0...（第1000位为1）
差异:   第1000位发生了变化
```

**如何推断出具体人的凭证被吊销：**

这里的关键在于索引号是写在VC里的，而VC可能在某个时间点被出示过。当Bob曾经向一个验证方出示过VC时，验证方知道Bob的VC索引是1000。如果这个验证方（或者任何见过这张VC的人）后来发现第1000位从0变成了1，他就可以确认"Bob的VC被吊销了"。

更严重的是，如果发行方将位串分段（例如每个部门一个位串），观察者甚至可以根据哪个位串发生变化来推断是哪个类别的凭证被吊销，进一步缩小范围。

---

### 风险3：吊销率泄露

**吊销操作在账本上留下哪些可观察痕迹：**

以Hyperledger Indy为例，其采用密码学累加器方案。每次吊销操作会在账本上产生以下记录：

- 一笔新的交易，包含更新后的累加器值
- 一个"witness delta"（见证差量），记录了本次被吊销的VC对应的素数因子

这个witness delta是公开的，因为所有持有有效VC的人都需要下载它来更新自己的见证值。它的存在本身就意味着"有一张VC被吊销了"。

再以BSL（位串状态列表）为例，每次吊销就是在位串中翻转一个bit，然后发布新版本。观察者可以：

- 通过计算新旧版本中值为1的bit数量之差，得知本次操作吊销了几张VC
- 通过记录每次发布的时间戳，计算出单位时间内的吊销速率

**外部观察者如何推断发证规模和吊销模式：**

**推断发证规模：** 在BSL中，位串的总长度是预先分配的，对应最大发证数量。已被分配（无论有效或已吊销）的索引数量可以通过某些旁信道推断。在Indy中，tails文件（尾部文件）的大小直接对应最大发证数量，该文件是公开下载的，文件大小即暴露了累加器的容量规划。

**推断吊销模式：** 如果一家公司每月固定在某几天发生吊销高峰（例如每月最后一天批量处理离职员工），观察者通过长期监测可以发现这种规律，从而推断该公司的人员管理模式，甚至预测未来的吊销时机。Schumm等方案中，由于每次吊销都产生单独一笔链上交易，吊销数量与交易数量是一一对应的，外部观察者直接数交易笔数就能得知精确的吊销数量。

---

### 风险4：状态索引关联攻击

**具体机制：**

在BSL中，Bob的VC里写明了"我的吊销状态在位串X的第1000位"。当Bob先后向医院和银行出示VC时，两个验证方都会读到同样的索引号1000。如果医院和银行互相交换信息（或被同一方监控），他们可以确认"查询第1000位的那个人，就是同时在我们两家机构活动的同一个人"，从而在两次不相关的交互之间建立联系，破坏了持有者本应享有的不可链接性。

---

### 风险5：见证更新过程中的隐私泄露

**具体机制：**

在累加器方案中，每当有VC被吊销，所有持有有效VC的人的见证值都会失效，需要向发行方或更新服务请求新的见证值。这个更新请求包含持有者的见证标识信息。发行方托管更新服务时，可以通过统计谁在什么时间请求了更新，来建立持有者的活跃度档案。即使由第三方托管，更新请求的时间模式本身也是元数据。

---

### 风险6：发行方活动信息泄露

这个已在风险3中详细展开，核心总结如下：账本上每一笔吊销交易都是时间戳可查的公开记录，通过统计交易频率可得吊销速率，通过累积交易数量可得历史吊销总量，通过分析交易内容中的delta信息可得单次吊销数量，最终使外部观察者能够重建出发行方的完整运营画像。

---

## 二、Prevoke和CRSet在具体流程中的隐私保护，以及缺失时的后果

### Prevoke的隐私保护流程

**保护PR1（防Phone Home）的具体流程：**

有保护的流程是：

1. 验证方收到持有者的VP
2. 验证方直接向以太坊智能合约查询布隆过滤器状态
3. 智能合约返回结果
4. 整个过程发行方完全不知情

如果没有这个保护，按照传统OCSP流程：

1. 验证方（雇主2）收到Bob的VC
2. 验证方向发行方（雇主1）发送查询请求，附带Bob的凭证ID
3. 雇主1的服务器记录下："雇主2在今天查询了Bob的凭证"
4. 雇主1得出结论：Bob正在向雇主2求职，可能即将离职
5. 后果：雇主1可能对Bob采取不利行动，例如降薪、边缘化，甚至提前解雇

**保护PR2（持有者匿名）的具体流程：**

有保护的流程是：

1. 发行阶段：智能合约只存储H(vcID)（哈希值），不存储vcID本身
2. 吊销阶段：发行方将默克尔树中Bob对应叶节点的值替换为H(随机字符串)，布隆过滤器中设置H(vcID)对应的位
3. 验证阶段：Bob生成BBS零知识证明，只向验证方披露布隆过滤器索引和默克尔树证明路径，不披露vcID本身，这样VC文件就没有得到共享

如果没有BBS保护，按照IOTA Identity的流程：

1. Bob出示VC，其中包含明文的位图索引，例如index=1000
2. 验证方看到index=1000
3. 注册表上第1000位从0变为1
4. 任何曾经见过Bob出示VC的人（包括验证方）都可以得知"Bob的VC被吊销了"
5. 后果：如果Bob的离职是因为违规行为，这个信息可能扩散到所有曾与Bob交互过的机构，对Bob的声誉造成不可控的损害

**保护PR3（隐藏吊销率）的具体流程：**

有保护的流程是：

1. 发行方需要吊销3张VC时，同时处理，一次性更新布隆过滤器和默克尔树
2. 智能合约只存储前z层默克尔树，外部只能观察到某个z层节点发生了变化
3. 由于一个z层节点对应2^(h-z)张VC，观察者只知道"1到2^(h-z)张VC"被吊销，无法得知精确数字

如果没有这个保护，按照Schumm方案：

1. 发行方每次吊销一张VC就发送一笔交易
2. 竞争对手持续监控链上交易
3. 发现某公司在某季度的最后一周突然出现大量吊销交易
4. 后果：竞争对手推断该公司正在进行大规模裁员，提前在人才市场上布局，或者向媒体透露该公司运营不稳定的信号

---

### CRSet的隐私保护流程

**保护发行方活动隐私的具体流程：**

有保护的流程是：

第一步，发行方设定容量nmax（例如100万），无论实际发了多少VC，过滤器始终按照100万容量填充。

第二步，每24小时，无论有没有新的吊销，发行方都重新生成一个全新的过滤器级联并发布到以太坊blob。新的过滤器包含：

- 实际有效VC的ID集合V
- 实际已吊销VC的ID集合R
- 大量随机填充数据，将R填充到2×nmax，将V填充到nmax

第三步，验证方下载最新的blob，在本地执行成员检查。

如果没有填充和定期发布：

- 攻击者观察到某公司在3月15日发布了新版本，与上一版本相比，过滤器中排除的元素增加了50个
- 攻击者得出结论：该公司3月15日吊销了50张VC
- 结合已知该公司发行的是员工身份证明，攻击者推断：该公司在3月15日处理了50名员工的离职
- 后果：如果这50人中包括核心技术人员，竞争对手可以针对性地进行挖角，或者投资者据此判断公司人才流失严重

**如果没有定期强制发布：**

攻击者观察到：

- 1月1日：发布了版本A
- 1月15日：发布了版本B（距离上次14天后）
- 1月16日：发布了版本C（距离上次仅1天后）

从发布时间间隔可以推断：1月15日到16日之间发生了紧急吊销事件（否则为何打破了之前的14天周期）。即使不知道吊销了谁，这个异常本身就是信息泄露。

**如果没有密码学哈希，仅用普通哈希：**

攻击者已知该公司所有员工的工号（例如从领英获取），可以穷举计算每个工号的哈希值，然后与过滤器中的哈希值比对，从而还原出哪些工号对应的VC已被吊销，直接获知哪些员工已经离职。使用密码学哈希函数（如SHA-256）使这种穷举在计算上不可行。

## 三、仍然存在的隐私泄露风险及解决方向

### 风险1：长期时序关联攻击

**具体的攻击过程是怎样的：**

CRSet每24小时强制发布一次新版本，这本身是为了隐藏吊销时机。但攻击者可以建立一个长期档案库，持续下载每一个版本。

虽然每个版本的过滤器内容因为随机填充而看起来完全不同，但攻击者掌握的信息会随时间积累。假设攻击者通过某种途径（例如在某次VC出示过程中）获得了一个特定vcID，他虽然无法直接反推哈希，但可以用这个已知的vcID去查询每一个历史版本的过滤器，观察它从某个时间点开始不再出现在有效集合中，从而精确定位吊销发生的时间窗口。

CRSet论文本身承认，由于数据是公开的，长期来看无法保证绝对隐私。如果未来哈希函数出现弱点，或者攻击者积累了足够多的已知vcID样本，暴力匹配就变得可行。

**为什么差分隐私可以解决这个问题：**

差分隐私的核心思想是：在发布的数据中加入经过数学校准的随机噪声，使得"包含某个元素"和"不包含某个元素"的输出在统计上无法区分。

具体应用到过滤器发布场景：每次发布时，不仅填充随机ID，还对过滤器中某些位进行随机翻转，翻转概率经过ε-差分隐私参数校准。这样即使攻击者持有某个vcID，用它查询连续100个版本，每次查询结果都带有统计噪声，无法确定到底哪个版本发生了真实的状态变化。

代价是引入了一定比例的假阳性（有效VC被误判为已吊销），需要在隐私预算ε和可用性之间权衡。

### 风险2：验证方侧的关联攻击

**攻击的具体建立过程：**

假设Alice持有一张大学颁发的学历VC。她在2023年1月向银行出示，在2023年6月向房东出示，在2024年3月向新雇主出示。

如果每次出示的VC中都包含相同的凭证标识符（例如BSL的固定索引号），那么三个验证方看到的是同一个标识符。如果这三个验证方之间存在数据共享（例如都接入了同一个身份验证服务商），或者同一个攻击者控制了其中多个验证方，他就可以建立一张时间线：

```
2023年1月：Alice在申请银行贷款
2023年6月：Alice在租房
2024年3月：Alice在找新工作
```

这张时间线完整地描绘了Alice的生活轨迹，即使她每次出示的都是同一张"匿名"凭证。

**BBS证明如何从根本上解决这个问题：**

BBS签名方案允许持有者从同一个原始签名派生出无限个零知识证明，每个证明在数学上与原始签名和其他派生证明完全无关。

具体流程是：

1. 发行方用私钥对Alice的VC签名，生成原始签名sig
2. Alice向银行出示时，用sig派生出证明π₁，π₁中包含一个由Alice随机生成的nonce，使得π₁对银行是全新的随机值
3. Alice向房东出示时，派生出π₂，π₂与π₁在数学上完全不相关
4. 即使银行和房东共享数据，他们也无法将π₁和π₂关联到同一个人

每次出示相当于使用一次性通行证，而不是每次都出示同一张身份证。

### 风险3：网络层元数据泄露

**泄露的具体内容和推断过程：**

即使CRSet的内容层完全私密，下载行为本身也是可观察的。以下是一个具体场景：

某医院作为发行方，在以太坊上发布了医生执照的吊销状态。全国有100家诊所（验证方）需要验证医生资质。

一个监控网络流量的攻击者可以观察到：

- 每个IP地址何时下载了哪个以太坊地址的blob数据
- 某诊所在某段时间内下载频率突然增加

从这些元数据可以推断：

- 哪些机构在积极验证医生资质（说明正在招聘）
- 某诊所突然停止下载（说明可能停业或换了系统）
- 下载频率的地域分布（说明哪个地区对某类凭证的需求最高）

这些信息虽然不涉及任何个人，但对发行方和验证方而言都是敏感的业务情报。

**PIR技术的工作原理：**

私有信息检索（Private Information Retrieval）允许用户从数据库中检索特定条目，而服务器无法得知用户检索了哪个条目。

最简单的PIR方案：服务器将数据分成n个分块，用户下载全部n个分块，在本地找到需要的那个。代价是需要下载全部数据，但服务器完全不知道用户感兴趣的是哪一块。

更高效的方案（如基于同态加密的PIR）：用户将查询请求加密后发给服务器，服务器在加密状态下执行检索并返回加密结果，整个过程服务器无法解密查询内容。

应用到VC场景：验证方发送加密的凭证ID查询，吊销数据服务器返回加密的状态结果，服务器日志中没有任何可识别的查询记录。

**Tor网络思路的具体应用：**

在Tor中，流量经过多个节点中转，每个节点只知道上一跳和下一跳，无法同时知道请求的来源和目的地。

应用到吊销数据下载：在验证方和以太坊节点之间引入一个由多方运营的中继网络。验证方通过随机路径下载blob数据，以太坊节点只看到中继节点的IP，中继节点只知道上下游，无人能同时掌握"谁下载了什么"的完整信息。

### 风险4：容量实例切换泄露

**泄露的具体机制：**

CRSet的每个实例都有一个预设容量nmax。当发行方创建第二个实例（新的以太坊地址）时，意味着第一个实例已经被填满或接近填满。

攻击者可以观察到：

1. 某发行方的第一个以太坊地址在时间T₁创建
2. 第二个地址在时间T₂创建
3. 从T₁到T₂的时间差，结合已知的nmax，可以计算出发证速率：nmax ÷ (T₂ - T₁) = 每天发证数量

如果发行方是一家公司，每天发证数量约等于每天入职人数，攻击者由此可以推断公司的规模增长速度。

**分散发证和随机切换的具体实现：**

分散发证方案：发行方同时维护多个实例，新的VC随机分配到其中一个实例，而不是等一个填满再开下一个。这样每个实例的填充速度都不规律，攻击者无法从单个实例的创建时间推断总体发证速率。

随机切换时机：即使某个实例还有剩余容量，发行方也按照泊松过程随机决定是否切换到新实例，使切换事件在时间上服从随机分布，打破与发证规模的确定性关联。

但这两种方法都有代价：维护多个实例增加了运营成本，随机切换导致部分实例容量浪费。

---

### 风险5：跨凭证关联攻击

**具体攻击场景：**

Bob持有来自同一家大学的两张VC：一张学历证明，一张在校期间的荣誉证书。这两张VC都指向同一发行方的吊销注册表。

当Bob向某雇主同时出示这两张VC时，验证方执行两次吊销状态查询：

- 查询1：学历证明的吊销状态
- 查询2：荣誉证书的吊销状态

在BSL方案中，这两次查询发生在极短的时间间隔内，且都指向同一个位串。如果验证方记录了查询日志，他可以发现这两个不同的索引号几乎总是同时出现在查询记录中。通过长期观察，验证方可以建立一个"同时出现对"的数据库，推断哪些凭证属于同一个持有者，即使他从未知道持有者的真实姓名。

更进一步，如果其中一张凭证因为某次泄露被关联到真实身份，另一张凭证也随之被去匿名化。

**群签名如何解决这个问题：**

群签名允许群成员（所有VC持有者）以群的名义签名，验证方只能验证签名者是某个合法群成员，但无法确定是哪一个，也无法关联同一人的不同签名。

具体应用：大学为所有毕业生建立一个签名群。Bob出示学历证明时，用群签名生成证明P₁；出示荣誉证书时，用同一群签名机制生成证明P₂。验证方可以验证P₁和P₂都来自合法的大学毕业生，但无法判断它们是否来自同一个人。

环签名的思路类似：Bob用自己的私钥加上其他群成员的公钥生成一个"环"，签名在环中任意一个成员的名义下，验证方无法确定真正的签名者。

### 风险6：量子计算威胁

**现有方案具体依赖哪些密码学假设，量子计算如何破坏它们：**

Prevoke使用BBS签名，其安全性基于BLS12-381椭圆曲线上的离散对数问题和配对假设。Shor算法可以在多项式时间内求解离散对数问题，一旦有足够规模的量子计算机（估计需要数千到数万个逻辑量子比特），BBS签名的安全性就完全崩溃。

Hyperledger Indy使用RSA累加器，其安全性基于大整数分解问题，同样是Shor算法的攻击目标。

CRSet使用SHA-256哈希，SHA-256面临的是Grover算法的威胁，Grover算法将暴力搜索的复杂度从O(2^n)降低到O(2^(n/2))，相当于将SHA-256的有效安全性从256位降低到128位。这在当前被认为仍然足够安全，但理论上需要迁移到SHA-512级别。

**格密码学的具体工作原理和应用：**

基于格（Lattice）的密码学的安全性依赖于在高维格中找到最短向量（SVP）或学习带误差（LWE）问题，目前没有已知的量子算法能高效求解这些问题。

在VC吊销场景中的具体应用方向：

基于格的零知识证明可以替代BBS签名中的椭圆曲线配对，实现后量子安全的选择性披露。基于格的累加器（如Libert等人提出的方案）可以替代RSA累加器，提供后量子安全的成员证明。代价是证明大小和计算开销比椭圆曲线方案大若干倍，但NIST后量子标准化进程（已于2024年完成首批标准）正在推动这一领域快速成熟。

STARK系统之所以具有后量子友好性，是因为它的安全性仅依赖抗碰撞哈希函数，而不依赖任何代数结构假设。Grover算法对哈希的威胁可以通过简单地增加输出长度来对抗，使其成为目前最现实的后量子零知识证明候选方案之一。

## 问题1：Prevoke中布隆过滤器的作用，以及只用默克尔树行不行

### 先理解默克尔树在Prevoke中的角色

根据论文，默克尔树累加器（MTAcc）存储的是**所有有效VC**的哈希值。每个叶节点对应一张VC的H(vcID)。当一张VC被吊销时，发行方将该叶节点的值替换为H(随机字符串)，使该VC的成员证明永久失效。

验证流程中，持有者需要向验证方提供一个默克尔树证明（MTAcc proof），证明自己的VC仍在有效集合中（即叶节点未被替换）。这需要持有者提供从自己的叶节点到根节点的完整路径，也就是**见证值（witness）**。

### 那如果只用默克尔树会发生什么

**第一个问题：见证值更新的代价极其高昂。**

论文中明确指出，每当MTAcc发生任何更新（无论是新增还是吊销），所有持有者的见证值（路径信息）都会失效，需要重新获取。原因是默克尔树的根哈希是由所有叶节点逐层哈希计算得出的，任何一个叶节点变化都会改变根哈希，导致所有依赖旧根哈希的见证值不再有效。

举个具体例子：如果发行方发出了10万张VC，其中有1张被吊销，那么另外99999张VC的持有者都需要更新自己的见证值。这是一个灾难性的通信开销。

**第二个问题：只用默克尔树，每次验证都需要进入"重量级"流程。**

默克尔树证明需要持有者提供O(log n)个哈希值（树的高度层数），验证方需要逐层重新计算根哈希来验证。对于10万张VC，树的高度约为17层，每次验证需要17次哈希计算和网络传输。这还不是最大的问题，最大的问题是**持有者必须持有最新的见证值**，而获取最新见证值需要联系发行方或从智能合约下载，这违反了PR1（防Phone Home）。

### 布隆过滤器的真正作用：作为高效的第一道过滤器

布隆过滤器在Prevoke中存储的是**所有已吊销VC**的哈希索引（注意：和默克尔树方向相反，一个存有效，一个存无效）。

布隆过滤器有一个关键特性：**永远不会产生假阴性（false negative）**。也就是说，如果一张VC确实被吊销了，布隆过滤器一定会返回"已吊销"。但它可能产生假阳性（false positive），即某张有效VC被误判为已吊销。

这个特性使得Prevoke可以设计出两阶段验证：

**阶段1（布隆过滤器检查）：**

- 验证方将持有者VC的布隆过滤器索引（bfIndexes）发送给智能合约
- 如果布隆过滤器返回"不在其中"→ 该VC**一定有效**，验证结束
- 如果布隆过滤器返回"在其中"→ 可能吊销，也可能是假阳性，进入阶段2

**阶段2（默克尔树证明）：**

- 只有阶段1返回"在其中"的VC才需要进入这一步
- 持有者提供最新的见证值，验证方验证默克尔树证明
- 如果证明有效 → VC确实有效（之前是假阳性）
- 如果证明无效 → VC确实已被吊销

**关键数字：** 论文设置假阳性率p=0.1，意味着在10万张有效VC中，只有约1万张会被误判进入阶段2。论文实验数据（图2）显示，阶段1验证时间仅13-16毫秒，而阶段2验证时间高达40-3200毫秒（取决于z值）。

所以布隆过滤器的核心价值在于：**让绝大多数有效VC（约90%）只需要经过一次快速的、无需更新见证值的阶段1检查就能完成验证**，极大地降低了见证值更新频率和总体计算开销。

### 布隆过滤器对PR3（隐藏吊销率）也有贡献

布隆过滤器中，多个不同的vcID可能映射到同一组位索引（因为哈希碰撞）。所以即使攻击者观察到布隆过滤器中某些位从0变为1，他也无法确定到底有几张VC被吊销，因为一次位变化可能对应1张VC，也可能对应多张VC（它们恰好哈希到同一个位置）。这是布隆过滤器概率性质带来的天然隐私保护。

## 问题2：CRSet解决的问题，"可计数隐私"是什么

### CRSet原文说它解决的核心问题

论文的原话是：

> "No existing solution protects issuers from leaking operational information such as the number or timing of revocations."

也就是说，现有所有方案（包括BSL、AnonCreds、Prevoke）都无法防止外部观察者推断出**发行方吊销了多少张VC**以及**什么时候吊销的**。CRSet的目标就是填补这个空白。


### "可计数隐私"：论文中的Chosen Count Indistinguishability

**直觉理解：**

设想一个游戏。攻击者（adversary）向挑战者（challenger）提交两个不同的场景：

- 场景A：某公司过去30天里，第1天发了1000张VC，第15天吊销了50张，第30天又吊销了30张
- 场景B：某公司过去30天里，第1天发了500张VC，第10天吊销了200张，第20天吊销了100张

挑战者随机选择场景A或场景B，按照选定的场景生成相应的过滤器数据，并把这些数据交给攻击者。攻击者看完数据后，需要猜测挑战者选择的是A还是B。

**如果吊销机制满足Chosen Count Indistinguishability，则攻击者猜对的概率不超过50%加上一个可忽略的误差。** 也就是说，从数据结构本身完全无法区分这两种场景，攻击者的最优策略就是随机猜，猜对的概率和抛硬币一样。

**为什么CRSet能满足这个定义：**

论文的证明分两步：

第一步（引理1）：论文定义了一个修改版的CRSet，叫做crset'，它完全忽略输入的有效VC集合V和吊销VC集合R，直接用纯随机数填满过滤器。对于这个crset'，攻击者猜对的概率显然是50%，因为输出与输入完全无关。

第二步（引理2）：论文证明真实的CRSet的输出与crset'的输出在计算上不可区分，因为CRSet使用的是密码学哈希函数（随机预言机模型下），真实vcID的哈希值和随机填充的哈希值在统计分布上完全相同，攻击者无法区分哪些哈希来自真实VC，哪些来自随机填充。

**关键机制：固定填充比例**

论文规定填充比例为：有效VC集合填充到nmax，吊销VC集合填充到2×nmax，比例固定为1:2。

为什么是1:2？论文解释：这是为了应对极端情况——如果发行方需要紧急吊销所有有效VC（例如密钥泄露），吊销集合的容量必须足以容纳所有曾经发出的VC。假设长期吊销率不超过50%，最坏情况下吊销集合需要是有效集合的2倍。

**固定比例的隐私含义：** 无论实际有1张还是100万张VC，过滤器的大小始终一样，存储的哈希数量始终一样，攻击者无法从数据结构的任何维度推断出真实数量。

---

**总结对比：**

| 机制     | Prevoke的布隆过滤器                         | CRSet的过滤器级联                                |
| ------ | ------------------------------------- | ------------------------------------------ |
| 存储内容   | 已吊销VC的哈希索引                            | 有效VC的ID（排除吊销VC）                            |
| 主要作用   | 快速过滤，减少默克尔树查询频率                       | 隐藏发行方的绝对和相对吊销数量                            |
| 隐私保护维度 | PR1（Phone Home）+ PR2（持有者匿名）+ PR3（吊销率） | 发行方活动隐私（Chosen Count Indistinguishability） |
| 是否交互   | 阶段2需要与发行方交互                           | 完全非交互                                      |
| 假阳性处理  | 进入默克尔树阶段2验证                           | 不存在假阳性问题（级联消除了假阳性）                         |

## 问题3. Prevoke 的吊销与验证流程：分场景讨论

在 Prevoke 中，核心是“本地布隆过滤器（BF）检查 + 链上 Merkle Tree（MT）存证”**。

### **场景 A：用户被吊销了**
	
1. **Issuer 操作：** 将该用户的 $vcID$ 通过 $k$ 个哈希函数映射到 BF 的 $k$ 个位，并将这些位从 `0` 变为 `1`。随后更新包含这些位的 MT 叶子节点（Chunks）以及根哈希。
        
2. **验证流程：** 验证方计算该用户的 $k$ 个索引。他在下载的最新 BF 中发现这 $k$ 位**全是 `1`**。同时MT也失效
        
3. **结论：** 验证方判定该凭证已失效，流程终止。
        
### **场景 B：用户没被吊销，但路径上有更新（由于他人被吊销）**
    
1. **持有者状态：** Bob 的凭证没变，但他持有的“见证值”（Merkle Path）因为其他人（比如 Alice）被吊销导致树根变了，旧路径失效。
        
2. **流程：** 
	- **第一步：** Bob 必须先从 Issuer 的“更新服务”或链上同步数据，获取**最新的见证值**（即新的兄弟节点哈希），确保能指向新的根哈希。
        
    - **第二步：** 验证方检查 BF，发现 Bob 对应的 $k$ 个位**至少有一个是 `0`**。
            
    - **第三步**：但为了防止 BF 的假阳性或 Issuer 作恶，验证方要求 Bob 提供 MT 证明。Bob 出示**刚更新的见证值**，证明这几个为 `0` 的位确实属于当前的合法树结构。
            
3. **结论：** 验证通过。
        
### **场景 C：用户没被吊销，且路径也没更新**
    
这种情况最简单。验证方查 BF 发现有 `0`，Bob 提供旧的见证值，算出来的根哈希依然与链上一致。

##  问题4. 关于“哈希单向性”与隐私泄露的质疑

 如果攻击者（如前雇主或竞争对手）**已经知道了**某个人的 $vcID$，即使存储的是 $H(vcID)$，攻击者依然可以不断重算哈希并去匹配撤销列表。

**Prevoke 解决这个问题的差别在于：**

- **不公开完整的 $H(vcID)$：** 在传统的列表方案中，撤销列表直接写着 `Revoked: H(vcID_1), H(vcID_2)...`。攻击者拿自己的目标哈希一比对就中了。
    
- **布隆过滤器的模糊性：** Prevoke 存储的是一个**位数组**（0101...）。
    
    - 即使你知道 Bob 的 $H(vcID)$ 对应索引 10, 50, 90。
        
    - 由于布隆过滤器的**哈希碰撞**，索引 10 变成 `1` 可能是因为 Bob 被吊销了，也可能是因为 Dave、Eva 或其他 100 个人中的某一个被吊销了。
        
    - **差别：** 存储 $vcID$ 或其哈希是**确定性泄露**；布隆过滤器提供的是**概率性掩护**。攻击者只能怀疑，不能确认。

## 问题5. CRSet 的历史数据比对（差分分析）

你提到的“下载历史数据并对比”在安全领域被称为**差分攻击（Differential Attack）**。

如果 CRSet 只是简单地添加 padding，确实会被历史对比破解。但 CRSet 的设计包含两个关键防护：

1. **完全重构（Full Regeneration）：** CRSet 并不是在旧的过滤器上“增加”数据，而是每 24 小时**使用全新的盐（Salt）或随机种子**重新生成整个过滤器。
    
    - **结果：** 昨天的过滤器里 Bob 的位置在索引 100；今天的过滤器里，因为 Salt 变了，Bob 的位置变成了索引 9982。
        
    - **后果：** 即使你下载了两个版本的 Blob 数据，它们看起来就像两串完全随机、毫无关联的二进制流。你无法通过“异或（XOR）”或对比来发现哪个位多了一个 `1`。
        
2. **恒定填充（Constant Padding）：**
    
    无论实际吊销了多少，它始终填充到固定的 $2 \times n_{max}$ 大小。配合第一点的“完全重构”，外部观察者看到的只是每 24 小时发布的一个**大小固定、内容全乱**的加密块。

**真正能解决这个问题的方向：**

如果要防止"已知vcID情况下的历史追踪"，需要引入**前向安全性**机制。例如定期为所有VC重新分配新的随机vcID，并通过安全信道通知持有者。这样即使攻击者掌握了旧的vcID，也无法在新版本的过滤器中追踪同一张VC。但这会显著增加系统复杂性，目前CRSet没有实现这一点，是一个开放的研究问题。

## 问题6：布隆过滤器的k个索引是什么意思，为什么全为1就代表被吊销

### 先理解布隆过滤器的基本工作原理

布隆过滤器本质上是一个很长的**位数组**，初始时所有位都是0。论文中，这个位数组的长度m和哈希函数数量k由两个参数决定：

```
m = -(r × ln p) / (ln 2)²
k = (m/r) × ln 2
```

其中r是预期吊销数量，p是假阳性率。

**一个具体的例子：** 假设m=100（位数组长度100），k=3（3个哈希函数H1、H2、H3）。

初始状态：

```
位置: 0  1  2  3  4  5  ...  50  ...  90  ...  99
值:   0  0  0  0  0  0  ...  0   ...  0   ...  0
```

### 插入一个元素（吊销操作）

当Bob的VC（vcID="Bob123"）被吊销时，发行方计算：

- H1("Bob123") = 10（第10位）
- H2("Bob123") = 50（第50位）
- H3("Bob123") = 90（第90位）

然后将这三个位置全部设为1：

```
位置: 0  1  2  ...  10  ...  50  ...  90  ...  99
值:   0  0  0  ...  1   ...  1   ...  1   ...  0
```

这就是论文中说的"bfIndexes = H1(vcID), H2(vcID), ..., Hk(vcID)"，k个索引就是k个哈希函数分别计算出的位置。

### 查询操作（验证时）

验证方收到持有者的VP后，同样计算这k个哈希：

- H1("Bob123") = 10
- H2("Bob123") = 50
- H3("Bob123") = 90

然后去布隆过滤器中检查这三个位置：

- 第10位 = 1 ✓
- 第50位 = 1 ✓
- 第90位 = 1 ✓

**三个位置全为1 → 判定Bob可能被吊销，进入阶段2。**

### 为什么全为1才代表（可能）被吊销

这里有一个关键的逻辑：

**如果某一位是0，则该VC一定没有被吊销。** 因为吊销操作会把k个位全部设为1，如果有任何一位还是0，就说明这张VC从来没有被插入过布隆过滤器，即从未被吊销。

**如果k个位全为1，则该VC可能被吊销（也可能是假阳性）。** 因为存在哈希碰撞的可能：

继续上面的例子，假设Carol的vcID也被吊销：

- H1("Carol456") = 10（和Bob一样！）
- H2("Carol456") = 33
- H3("Carol456") = 77

插入Carol后：

```
位置: 0  1  ...  10  ...  33  ...  50  ...  77  ...  90  ...
值:   0  0  ...  1   ...  1   ...  1   ...  1   ...  1   ...
```

现在假设Dave的VC没有被吊销，但：

- H1("Dave789") = 33（碰撞！Carol占了这位）
- H2("Dave789") = 50（碰撞！Bob占了这位）
- H3("Dave789") = 77（碰撞！Carol占了这位）

查询Dave：第33、50、77位全为1 → 误判Dave被吊销，但实际上Dave是无辜的。这就是假阳性。

### 这与Prevoke的隐私保护有什么关系

这里回到问题4的核心。布隆过滤器存储的不是"Bob被吊销了"这个明确信息，而是k个位置上的1。攻击者即使知道Bob的vcID，计算出第10、50、90位，看到这三位全是1，也**无法确认**是Bob导致这些位变成1，还是其他某个人的vcID恰好也映射到了这些位置。这就是布隆过滤器提供的概率性掩护。

## 问题7：CRSet每24小时更新，吊销是否不及时

### CRSet确实存在最长24小时的吊销延迟

论文的设计是：发行方每24小时重新生成并发布一次过滤器级联。这意味着：

如果某员工在早上9点被解雇，发行方立刻执行了吊销操作，但下一次发布要到第二天同一时间。在这24小时内，该员工的VC在CRSet中仍然显示为有效，他依然可以用这张VC通过任何验证。

论文对这个问题的态度是：发行方可以根据自己的安全需求**缩短发布间隔**，例如改为每小时发布一次，代价是更高的计算和gas费用。24小时只是论文给出的一个默认参数，不是硬性限制。

### 与其他方案相比，这个延迟是否可以接受

这取决于具体场景：

**可以接受的场景：** 学历证明、职业资格证书等长期有效的凭证，24小时的延迟几乎没有实质影响。一张大学文凭不需要分钟级别的吊销响应。

**不可接受的场景：** 员工门禁卡、医生当日行医资格等需要即时失效的凭证，24小时延迟是严重缺陷。

### 论文没有完全解决这个问题，这是CRSet的真实局限

CRSet论文的核心目标是解决**发行方活动隐私**，而不是最优化吊销实时性。论文自己也没有声称解决了实时吊销问题。

但这里有一个更深的矛盾值得注意：**实时吊销与发行方隐私之间存在根本性张力。**

如果CRSet改为每次有吊销就立刻发布，那么发布时机本身就泄露了信息（某时刻发生了吊销事件），这正是论文强制每24小时发布来规避的问题。要实现真正的实时吊销同时保护发行方隐私，理论上需要一种机制：在吊销发生时立刻生效，但在外部观察者看来，这次发布与正常的定时发布无法区分。

目前没有方案完美解决这个矛盾，这是一个真实存在的开放研究问题。Prevoke通过智能合约的实时写入实现了即时吊销（发行方随时可以更新链上状态），但代价是放弃了对发行方活动量的完全隐藏。CRSet通过定时批量发布实现了发行方活动隐私，但代价是引入了吊销延迟。两者在这个维度上做出了不同的权衡选择。

## 问题8：Prevoke和CRSet是否解决了状态索引关联攻击

### Prevoke的情况

**先看Prevoke中持有者出示VC时披露了什么。**

根据论文Algorithm 4，持有者生成的VP包含：

```
vp = (π, nonce, bfIndexes, H(vcID), C)
```

其中bfIndexes就是k个布隆过滤器索引，H(vcID)是vcID的哈希值。

这里有一个严重的问题：**bfIndexes是固定的**。

因为bfIndexes = H1(vcID), H2(vcID), ..., Hk(vcID)，这k个索引完全由vcID决定，而vcID是不变的。Bob每次向任何验证方出示VC时，bfIndexes始终是同一组数字，例如始终是{10, 50, 90}。

这意味着：

- Bob向医院出示VC → 医院看到bfIndexes = {10, 50, 90}
- Bob向银行出示VC → 银行看到bfIndexes = {10, 50, 90}
- 医院和银行交换信息 → 确认是同一个人

**Prevoke没有解决状态索引关联攻击。**

但论文使用了BBS零知识证明，这里有一个微妙之处需要说清楚。BBS证明本身是不可链接的，也就是说π₁和π₂看起来完全不相关。但问题在于，VP中除了π之外还包含了**明文的bfIndexes**，这个明文信息破坏了BBS证明本来能提供的不可链接性。

论文对这个问题是有意识的。在结论部分，论文提到：

> "it is currently possible for a verifier to indefinitely monitor the revocation status of a VC it has received from a holder"

这说明作者知道验证方可以用固定的bfIndexes持续追踪持有者的状态，但将这个问题留作了future work，并没有在Prevoke中解决。

**所以Prevoke存在状态索引关联攻击的漏洞，且论文自己承认了这一点。**

---

### CRSet的情况

CRSet的设计与BSL和Prevoke有根本性的不同。

在CRSet中，每张VC被分配一个随机的**revocation ID**，论文原文：

> "Every VC is randomly assigned a revocation ID, which we will refer to as just 'ID', from the large universe U = {0,1}²⁵⁶. It is entirely unrelated to the VC's id field."

这个revocation ID是256位随机字符串，与任何外部可识别信息无关。

**但关键问题是：这个revocation ID在验证时是否被披露？**

根据论文Algorithm 2，验证方执行的操作是：

```
输入：VC的revocation id，以及过滤器级联和salt
输出：True（有效）或False（已吊销）
```

验证方需要知道revocation ID才能执行查询。而revocation ID存储在VC的credentialStatus字段中，是VC的一部分，持有者出示VC时会包含这个字段。

这意味着：

- Bob向医院出示VC → 医院看到revocation ID = "a3f9...d2"
- Bob向银行出示VC → 银行看到revocation ID = "a3f9...d2"
- 医院和银行交换信息 → 确认是同一个人

**CRSet同样没有解决状态索引关联攻击。**

但CRSet的设计目标从一开始就不是解决持有者的关联攻击问题。论文Table 1明确列出了各方案的隐私保护维度：

|方案|Subject Privacy|RP Privacy|Issuer Privacy|
|---|---|---|---|
|BSL|PIR|No|No|
|Prevoke|Yes|Yes|Rate only|
|CRSet|Yes|Yes|**Yes**|

CRSet的Subject Privacy标注为Yes，但这里的"Yes"指的是**吊销状态本身不被泄露给非相关方**，而不是跨验证方的不可链接性。

---

### 根本原因：这是一个系统设计层面的矛盾

状态索引关联攻击的根本原因是：**验证方必须知道如何查询某张VC的吊销状态，而这个查询信息（无论是索引号还是revocation ID）本身就可以作为持有者的标识符。**

只要验证方需要拿着某个固定标识符去查询吊销状态，这个标识符就可以被用来跨会话关联同一个持有者。这不是Prevoke或CRSet的实现问题，而是现有吊销范式的结构性缺陷。

## 问题9：实时吊销与发行方隐私之间的矛盾，有没有解决办法

这个矛盾的本质是：

- **实时吊销**要求：吊销发生时立刻发布新数据
- **发行方隐私**要求：发布行为本身不携带"此时发生了吊销"的信息

这两个要求在直觉上是对立的。但我认为有几个方向值得思考，尽管都有代价。

### 方向一：承诺-揭示机制（Commit-Reveal）

**基本思路：**

将吊销分为两个阶段。

第一阶段（承诺阶段）：发行方在吊销发生时，立刻向链上提交一个承诺值C = Hash(吊销数据 || 随机盐)。这个承诺值本身不包含任何可识别信息，外部观察者看到的只是一个哈希值，无法推断发生了什么。

第二阶段（揭示阶段）：按照固定时间表（例如每24小时），发行方统一揭示这段时间内所有承诺的真实内容，并更新过滤器。

**效果：**

- 实时性：吊销在法律或业务层面立刻生效（链上有承诺记录）
- 隐私：揭示阶段仍然是批量的，外部观察者无法从揭示内容推断哪些承诺对应哪次吊销事件

**问题：** 在揭示之前，验证方无法验证最新的吊销状态，吊销的实际生效仍然有延迟。这只是把延迟从"技术延迟"转化为"法律意义上已吊销但技术上尚未可验证"，对于实时访问控制场景依然无效。

---

### 方向二：双轨制吊销（紧急通道+常规通道）

**基本思路：**

区分两类吊销场景，使用不同的机制处理：

**常规吊销**（例如员工正常离职）使用CRSet的24小时批量发布机制，完全保护发行方隐私。

**紧急吊销**（例如密钥泄露、欺诈行为）使用一个独立的紧急吊销注册表，实时生效，但代价是这个注册表的更新是可观察的。

为了降低紧急通道的信息泄露，可以对紧急通道引入**虚假吊销流量**：发行方定期向紧急注册表提交随机生成的假吊销条目，使真实的紧急吊销淹没在噪声中。

**效果：**

- 紧急场景实现实时吊销
- 常规场景保持完整的发行方隐私
- 紧急通道的真实吊销事件被虚假流量部分掩盖

**问题：** 虚假流量本身有成本，且如果攻击者能够区分真实和虚假的紧急吊销（例如通过其他旁信道），噪声保护就会失效。此外，这个方案暴露了哪些吊销属于"紧急"类别，本身就是一种信息泄露。

---

### 方向三：结合零知识证明的状态承诺

**基本思路：**

这是目前学术上最有潜力但也最复杂的方向。

发行方维护一个私有的吊销状态，每次更新时生成一个零知识证明，证明"新的过滤器是旧的过滤器经过合法更新得到的，且更新操作符合协议规则"，但不揭示更新了什么内容。

发布时，链上只存储新的过滤器和对应的零知识证明。验证方可以验证过滤器的合法性，但无法推断更新内容。

**效果：**

- 可以更频繁地发布更新（甚至实时），因为每次发布的内容在零知识证明保护下不泄露更新细节
- 发行方隐私由密码学而非时间批量来保证

**问题：** 这需要为布隆过滤器或过滤器级联的更新操作设计专用的零知识电路，计算开销极大。目前还没有成熟的方案将ZK证明与CRSet这类结构结合起来。这是一个真实存在的开放研究问题，结合了ZK-SNARK/STARK和隐私保护吊销两个领域，是值得深入研究的方向。

---

### 总结

|方向|实时性|发行方隐私|实现复杂度|当前成熟度|
|---|---|---|---|---|
|现有CRSet|❌ 24小时延迟|✅ 完整|低|已实现|
|现有Prevoke|✅ 实时|⚠️ 部分|中|已实现|
|承诺-揭示|⚠️ 法律实时/技术延迟|✅ 较好|中|未实现|
|双轨制|✅ 紧急场景实时|⚠️ 部分牺牲|中|未实现|
|ZK状态承诺|✅ 理论实时|✅ 完整|极高|研究阶段|
